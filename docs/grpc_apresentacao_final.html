Introdução

Arquiteturas de microsserviços tornaram a comunicação entre serviços um fator crítico de desempenho.

Cenário
📈
Crescimento dos microsserviços
Sistemas modernos são compostos por dezenas de serviços independentes que precisam se comunicar com eficiência.
Problema
⚠️
Limitações do REST interno
Em comunicação interna, REST gera latência extra, sobrecarga de JSON e falta de contrato formal entre serviços.
Solução
⚡
A proposta do gRPC
Comunicação binária, rápida e com contrato forte — desenvolvido pelo Google para resolver exatamente esse cenário.
"Em sistemas distribuídos modernos, a forma como os serviços se comunicam é tão importante quanto o que eles fazem."

O que é RPC
RPC (Remote Procedure Call) é exatamente esse garçom — um mecanismo que permite executar funções em outro computador da rede como se fossem locais. O desenvolvedor faz a chamada e recebe o resultado. O resto é invisível.
O que fica oculto
Comunicação pela rede
Serialização e desserialização dos dados
Gerenciamento de conexões e protocolos
Tratamento de erros de rede
Resultado
O desenvolvedor foca no que quer fazer, não em como a comunicação acontece.

WebSocket vs gRPC

Sem gRPC — WebSocket~16 linhas
const ws = new WebSocket('ws://localhost:8080');

ws.onopen = () => {
  // serializar manualmente
  const payload = JSON.stringify({
    user: 'Viviane',
    message: 'Olá!',
    time: Date.now()
  });
  ws.send(payload);
};

ws.onmessage = (e) => {
  // desserializar manualmente
  const msg = JSON.parse(e.data);
  renderMessage(msg);
};

ws.onerror = (err) => handleError(err);
O dev gerencia serialização, protocolo, erros e estrutura dos dados na mão. Sem contrato formal.
Com gRPC~7 linhas
const stream = client.Chat();

// enviar — só isso
stream.write({
  user: 'Viviane',
  message: 'Olá!',
  time: '10:32'
});

// receber — só isso
stream.on('data', (msg) => {
  renderMessage(msg);
});

// serialização: invisível
// protocolo: invisível
// erros de rede: invisível
Estrutura garantida pelo .proto. Serialização binária automática. O dev foca só na lógica.

Serialização de dados

Protocol Buffers

O que é
Protobuf é o mecanismo de serialização utilizado pelo gRPC. Em vez de transmitir texto legível como o JSON, ele converte os dados em formato binário compacto — menor, mais rápido e mais eficiente.
Por que é mais rápido
JSON transmite
{ "nome": "Viviane" }
// a chave "nome" vai como texto
Envia o nome do campo inteiro em cada mensagem — redundante e pesado.
Protobuf transmite
string nome = 1;
// só o ID 1 + o valor binário
Usa apenas o identificador numérico do campo. Muito menos bytes.

Contratos de serviço

Arquivo .proto

O contrato formal entre cliente e servidor — define tudo que o serviço oferece.

Exemplo do projeto
syntax = "proto3";
package chat;

service ChatService {
  rpc Login(LoginRequest)
      returns (LoginResponse);

  rpc Chat(stream ChatMessage)
      returns (stream ChatMessage);
}

message ChatMessage {
  string user = 1;
  string message = 2;
  string time = 3;
}
Componentes
syntax
Versão do Protobuf
Define a versão da linguagem. Hoje usamos proto3, mais simples e moderno.
service
Definição do serviço
Agrupa todos os métodos remotos disponíveis — é o "cardápio" do servidor.
rpc
Método remoto
Cada chamada disponível, com seu tipo de entrada e saída. O stream indica streaming bidirecional.
message
Estrutura de dados
Define os campos transmitidos. Os números (1, 2, 3) são IDs de serialização — nunca alterar em produção.
Protocolo de transporte

HTTP/2 — por que importa

HTTP/1.1 — o problema
Uma requisição por vez
Cada requisição precisa esperar a anterior terminar. Conexões paralelas consomem recursos excessivos.
A
B
C
B espera A · C espera B
Uma conexão por requisição
Cada chamada abre e fecha sua própria conexão TCP — overhead alto, latência acumulada.
HTTP/2 — a solução
Multiplexação — tudo ao mesmo tempo
Múltiplas requisições trafegam em paralelo numa única conexão TCP persistente.
A
B
C
D
A, B, C, D simultâneos · mesma conexão
📦
Compressão de cabeçalhos
Cabeçalhos repetitivos são comprimidos automaticamente, reduzindo o tráfego.
🌊
Streaming nativo
Dados fluem continuamente sem abrir novas conexões — base dos 4 tipos do gRPC.
Padrões de comunicação

Os 4 tipos do gRPC

01
Unary
Cliente──→Servidor
Cliente←──Servidor
1 requisição, 1 resposta. Modelo mais simples — como uma função tradicional.
uso: login, busca de dados
rpc Login(Req)
  returns (Res);
02
Server Streaming
Cliente──→Servidor
1 requisição, várias respostas. O servidor empurra dados continuamente.
uso: monitoramento, sensores
rpc Watch(Req)
  returns (stream Res);
03
Client Streaming
Cliente←──Servidor
Várias mensagens, 1 resposta. O servidor processa tudo e responde ao final.
uso: upload de arquivos
rpc Upload(stream Req)
  returns (Res);
04
Bidirectional Streaming
Cliente e servidor trocam mensagens simultaneamente. Conexão persistente.
★ usado no nosso chat
rpc Chat(stream Msg)
  returns (stream Msg);
